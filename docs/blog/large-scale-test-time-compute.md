# 大规模测试时计算的影响

> 原文：Noam Brown / @polynoamial, [Implications of Large-Scale Test-Time Compute](https://x.com/polynoamial/status/2064210146558136827)  
> 说明：以下为中文完整翻译，尽量保留原文结构与语气；文中配图为原文图片。

**tl;dr：随着 LLM 变得更强，基准测试表现越来越成为测试时计算量的函数。事实上，我们很可能并不知道现代 LLM 的能力上限在哪里，因为测量这个上限太昂贵了。我们应该改变 LLM 评测方式，把性能与 token、成本或时间之间的关系测出来。**

GPT-5.5 发布当天，最初的反应是怀疑。基准数字确实更好，但好得并不多：

![](/images/large-scale-test-time-compute/01.jpg)

然而，几个小时之内，等人们有时间真正上手这个模型之后，事情就变得清楚了：相比 GPT-5.4，它是一次阶跃式提升。经典的“基准表格”显然没有讲完整个故事。为什么会这样？

当我们把 token 放到 x 轴上，比较 GPT-5.5 和 5.4 时，原因就更清楚了：

![左图：在一个网络安全评测上，如果以各自“最大”测试时计算量来衡量，5.5 的表现看起来并没有比 5.4 好太多。右图：在另一个网络安全评测上，一旦我们控制 token/成本/延迟，就能清楚看到 5.5 比 5.4 强得多。](/images/large-scale-test-time-compute/02.jpg)

左图：在一个网络安全评测上，如果以各自“最大”测试时计算量来衡量，5.5 的表现看起来并没有比 5.4 好太多。右图：在另一个网络安全评测上，一旦我们控制 token/成本/延迟，就能清楚看到 5.5 比 5.4 强得多。

GPT-5.5 并不是在和 5.4 相同的 token 预算（或美元预算）下被评测的。一旦我们控制测试时计算量，5.5 看起来就比 5.4 强得多。

我经常和人讨论这个问题，他们会问：为什么我们不干脆用一个评测框架，不断增加测试时计算，直到性能达到平台期？问题在于，从经验上看，这个平台期非常远。有时在实际可承受的预算内，我们甚至根本看不到平台期。下面是 [@karpathy](https://x.com/@karpathy) 的 autoresearch 实验，性能在数百次实验之后仍然继续提升：

![](/images/large-scale-test-time-compute/03.jpg)

这里还有 @AISecurityInst 的网络安全评测，Mythos 和 GPT-5.5 的表现即使在超过 1 亿 token 之后，仍在快速改善：

![](/images/large-scale-test-time-compute/04.jpg)

注意，对于更强的模型，性能随时间提升的幅度也更强。看起来很可能是：随着模型变强，它们在更长时间跨度上运作的效率也会更高。平台期被推得更远，甚至可能消失。

因此，我认为评测模型的正确方式，是画出“性能 vs 测试时计算”的曲线，并把 token、成本或真实耗时中的某一个放在 x 轴上。已经有一些基准开始朝这个方向走。例如，ARC-AGI 衡量的是分数与成本之间的关系。

![](/images/large-scale-test-time-compute/05.jpg)

另一个合理选择，是设定一个明确的 token/时间/成本预算，并把它告知模型。这类似人类在 SAT 或国际数学奥林匹克竞赛这样的场景中接受评测的方式。

不同的 x 轴各有取舍。token 不能在模型之间直接比较，因为分词器、速度和每 token 成本都不同。美元成本取决于批处理、硬件利用率等实现细节，因此成本和延迟可以相互权衡。最后，真实耗时也不是完美度量，因为 best-of-N 这样的多智能体技术可以扩大测试时计算，而不显著增加延迟。尽管如此，这些曲线中的任何一种，都比单个标量更有信息量。

## 对 AI 准备工作的影响

在前沿模型发布之前，实验室通常会评估网络安全、生物安全以及其他滥用风险。如果某个模型跨过了能力阈值，那么发布可能会被推迟，直到缓解措施到位。但如果能力是推理计算量的函数，那么安全评测应该在什么样的推理预算下运行？

实践中，大多数用于模型发布的安全评测并不考虑模型使用了多少推理计算。Gemini 3 Deep Think 的发布以及由此引发的争议，是一个很有用的例子。

Gemini 3 Deep Think 发布时，它的基准分数远高于此前的模型。然而，与它一同发布的，并没有评估其风险的模型卡。

![](/images/large-scale-test-time-compute/06.jpg)

这引发了 AI 安全社区中一些人的愤怒。

![](/images/large-scale-test-time-compute/07.jpg)

在我看来，对 DeepMind 这次发布的批评错过了更深层的问题：AI 实验室和安全组织在评估模型发布时，并没有持续一致地把测试时计算考虑进去。

Deep Think 看起来很可能是对其他已有系统卡模型的脚手架封装。外部任何人很可能都可以复现这样的脚手架。换句话说，看起来很可能是：只要有人愿意为 Deep Think 那种规模的推理付费，通过把一堆模型查询脚手架化地组合起来，Deep Think 的能力本来就已经可以获得。Deep Think 只是让普通用户更方便地使用这种能力。

在我看来，真正应该引发愤怒的，是 Gemini 3 以及其他模型发布时，它们的系统卡没有把基准表现作为测试时计算量的函数来测量。在我理想中的世界里，模型评测应该大致长这样：

![](/images/large-scale-test-time-compute/08.jpg)

一个有组织的国家级行为体，可以把超过 1000 万美元的推理计算投入到单个任务上。但评测一个模型通常涉及数千次、甚至数百万次 rollout，所以如果每一次 rollout 都用这么高的计算预算来评测，那是不现实的。幸运的是，性能似乎会随着所施加的推理计算量以某种可预测的方式扩展。因此，我们可以在相对较低的推理预算下评测，然后外推（并带上不确定性）更高预算下可能具备的能力。

长周期评测会引入一些复杂性，而这些复杂性不一定总能通过从较小预算外推来解决。举例来说，可能只有真正把一个 AI agent 运行一年，才能有把握地评估它在一年时间跨度上的失调风险。AI 实验室可能很快会发现自己处在一种奇怪的位置：它们的 agent 可运行的时间跨度，超过了新模型的开发周期。到了那时，如果不推迟模型发布，可能就无法在发布前完成一个模型在其最大运行寿命范围内的评测。

## 具体建议

具体来说，我给 AI 社区的建议如下：

AI 实验室应该在发布新模型时，公布其基准表现，并把 token、成本或时间放在 x 轴上。至少，实验室应该报告达到某个标量基准结果时使用的推理预算。

基准测试应该在排行榜上追踪推理使用量，或者设置明确的 token/成本/时间预算。许多基准已经朝这个方向转变，但这还不是标准实践。

Preparedness Frameworks（准备框架）和 Responsible Scaling Policies（负责任扩展政策，RSP）在判断一个模型是否跨过安全阈值时，应该明确把推理计算纳入考虑。此外，评测应该估计多个推理预算下的能力，包括从较小预算运行结果出发、带有明确不确定性的外推。

如果你关注我已经有一段时间，这整篇文章可能看起来并没有什么新东西。自 2024 年 9 月 o1 发布以来，我们就已经知道，推理模型的表现会随着更多推理计算而扩展。

然而，将近两年之后，前沿 AI 实验室在发布新模型时，仍然普遍报告单一数字的基准结果；当一个脚手架系统使用 100 倍推理预算取得更好表现时，AI 安全组织仍然会感到惊讶；而 Preparedness Frameworks 和 RSP 在判断一个模型是否达到关键能力水平时，也仍然经常忽略推理计算使用量。

最新模型比以往任何时候都更能利用测试时计算，把性能平台期推得更远。如果这一趋势继续下去——而我完全预计它会继续——那么每经历一个模型发布周期，不考虑推理计算使用量的基准分数都会变得更没有信息量。正因为如此，现在是时候把推理预算视为能力测量和安全政策中的一等组成部分了。
