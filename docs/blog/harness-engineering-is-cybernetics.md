# 驾驭工程即控制论

> 原文：[Harness Engineering Is Cybernetics](https://x.com/odysseus0z/status/2030416758138634583) · 作者：George (@odysseus0z) · 2026年3月7日

---

读 OpenAI 的 [harness engineering 文章](https://openai.com/index/harness-engineering/)时，我一直有种说不清的感觉。然后突然想通了：我以前见过这个模式。不是一次——是三次。

第一次是 1780 年代 [James Watt 的离心调速器](https://en.wikipedia.org/wiki/Centrifugal_governor)。在它出现之前，一个工人站在蒸汽机旁边手动调节阀门。有了它之后，一个加重飞球机构自动感知转速并调节阀门。工人没有消失。工作变了：从转动阀门变成设计调速器。

第二次是 [Kubernetes](https://kubernetes.io/docs/concepts/architecture/controller/)。你声明期望状态——三个副本、这个镜像、这些资源限制。控制器持续观察实际状态。当两者偏离时，控制器进行调和：重启崩溃的 Pod、扩缩副本、回滚失败的部署。工程师的工作从重启服务变成了编写系统据以调和的 spec。

第三次是现在。OpenAI 描述了不再写代码的工程师。他们设计环境、构建反馈回路、编纂架构约束——然后 agent 来写代码。[五个月内一百万行代码](https://openai.com/index/harness-engineering/)，没有一行是手写的。他们称之为"harness engineering（驾驭工程）"。

每次都是同一个模式。[Norbert Wiener](https://en.wikipedia.org/wiki/Cybernetics:_Or_Control_and_Communication_in_the_Animal_and_the_Machine) 在 1948 年给它命了名：**控制论**（cybernetics），来自希腊语 κυβερνήτης——舵手。你不再转动阀门。你掌舵。

每次这个模式出现，都是因为有人构建了足够强大的传感器和执行器，在那个层面闭合了反馈回路。

## 为什么代码库一直是最后的堡垒

代码库有反馈回路，但只在较低层面。编译器在语法层面闭合回路。测试套件在行为层面闭合回路。Linter 在风格层面闭合回路。这些都是真正的控制论控制——但它们只在可以被机械检查的属性上运作。能编译吗？通过测试了吗？遵守规则了吗？

这些之上的一切——这个改动适合系统架构吗？这是正确的方法吗？这个抽象会在代码库增长时产生问题吗？——没有传感器，也没有执行器。只有人类能在那个层面同时完成两件事：判断质量和编写修复。

**LLM 同时改变了这两端。** 它们能在人类曾垄断的层面感知——也能在同一层面行动：重构模块、重新设计不一致的接口、围绕真正重要的契约重写测试套件。反馈回路第一次可以在重要决策层面闭合。

但闭合回路是必要条件，不是充分条件。瓦特的调速器需要调教。Kubernetes 控制器需要正确的 spec。而在你代码库上工作的 LLM 需要更难提供的东西。

## 校准传感器和执行器

让基本反馈回路运转起来——agent 能跑的测试、给出可解析输出的 CI、指向修复方案的错误信息——这是入场门票。[Carlini 在让 16 个并行 agent 构建 C 编译器时展示了这一点](https://www.anthropic.com/engineering/building-c-compiler)：[极其简单的 prompt](https://github.com/anthropics/claudes-c-compiler/commit/a28ff299)，但精心设计的测试基础设施。"我大部分精力都花在了设计 Claude 周围的环境上——测试、环境、反馈。"

更难的问题是用你系统特有的知识来校准传感器和执行器。这是大多数人卡住的地方，也是他们怪罪 agent 的地方。"它一直做错事。它不理解我们的代码库。"

**诊断几乎总是错的。** agent 失败不是因为缺乏能力。它失败是因为它需要的知识——在你的系统里"好"意味着什么、你的架构奖励哪些模式、回避哪些——被锁在你脑子里，你没有把它外化出来。

Agent 不靠耳濡目染学习。如果你不写下来，agent 在第一百次运行时犯的错跟第一次一模一样。

这项工作就是让你的判断对机器可读。描述实际分层和依赖方向的架构文档。内置修复指令的自定义 linter。编码了你团队品味的黄金准则。OpenAI [发现了完全相同的问题](https://openai.com/index/harness-engineering/)：他们每周五花 20% 的时间清理"AI 垃圾"——直到他们把标准编码进了 harness 本身。

## 唯一的出路

这要求的那些实践——文档、自动化测试、编纂的架构决策、快速反馈回路——一直都是正确的。过去三十年写的每一本工程书都推荐它们。大多数人跳过它们，因为跳过的代价缓慢而分散：质量逐渐下降、新人上手痛苦、技术债悄无声息地复利增长。

**Agent 工程让代价变得极端。** 跳过文档，agent 就会忽视你的规范——不是在一个 PR 上，而是在每一个 PR 上，以机器速度，全天候。跳过测试，反馈回路就根本无法闭合。跳过架构约束，漂移的复利速度超过你修复的速度。

这里有个陷阱：**如果 agent 不知道"干净"是什么样子，你就无法用 agent 来清理混乱。** 没有校准，制造问题的机器也解决不了问题。

实践没有变。忽视它们的惩罚已经变得不可承受。

**生成-验证不对称性**——[P vs NP](https://en.wikipedia.org/wiki/P_versus_NP_problem) 背后的直觉，被 [Cobbe 等人对 LLM 做了实证验证](https://arxiv.org/abs/2110.14168)——指出了这一切的走向。生成一个正确的解决方案比验证一个要难。你不需要在实现上超过机器。你需要在评估上超过它：规定"正确"是什么样子，识别输出何时偏了，判断方向是否正确。

设计瓦特调速器的工人没有回去转阀门。不是因为他们不能。而是因为那已经没有意义了。
