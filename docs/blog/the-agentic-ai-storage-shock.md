# Agentic AI 的存储冲击

> 原文作者：Ben Bajarin · The Diligence Stack

本周早些时候，我们参加了 Dell Tech World。与 Dell 高管、客户和一线实践者交流后，一个非常明确的体会是：企业数据层正在重新回到 AI 讨论的中心。只要理解工作流正在如何变化，就会发现 Agentic AI 让这一点变得不可避免。Claude Code/Cowork 和 Codex 是很有用的早期信号，因为它们展示了当 AI 不再只是云端聊天界面，而是获得对工作文件系统的读写权限后会发生什么。一旦模型能够在项目环境内部运行，工作流留下的就不只是最终输出。它会形成一条持久记录：工作是如何完成的、哪些上下文影响了它、结果被接受前发生了哪些改动。

这也是企业正在为更广泛工作流所准备的同一种转变。Agent 需要的不只是模型访问权限或应用界面。它们需要可检索、可授权、可验证、可执行、可写回且可保留的企业上下文。这一要求会给整个数据栈带来压力，从存储、检索，到治理、可观测性，再到控制数据访问的软件层，都会被重新审视。

理解这场转变的一个有效方式，是把它看成从“冷数据”到“温数据”的迁移。冷数据被存储、保留、归档，或者在技术上可以访问，但并不一定已经准备好在实时工作流中使用。它可能存在于文档、旧工单历史、文件共享、应用数据库、合规归档、邮件线程或数据湖里。人类用户通常可以通过搜索、解释、询问他人或依靠判断力来绕过这些碎片化问题。但 Agent 不能稳定地做到这一点，除非这些上下文被表示为系统能够检索、授权、验证并据此行动的形态。

Agentic AI 会把更多数据推入一个温热的操作层。温数据不一定是最热的交易数据，但它足够接近具体工作，因此可以派上用场。它被索引、被治理、具备权限感知能力、可观测，并且足够新，可以支持行动。客服 Agent 需要历史工单、产品文档、权益规则、客户历史和审计路径。采购 Agent 需要供应商条款、运输数据、审批阈值、异常历史，并能写回记录系统。编码 Agent 需要代码仓库、依赖、测试日志、构建产物和安全扫描结果。在每一种情况下，Agent 都不只是在搜索信息。它是在利用企业上下文完成工作。

这一观察会改变我们对存储及其上层软件层的理解。在 Agentic 企业中，数据既要被经济地存储、保护和恢复，也要可用于检索、复用、治理和审计。更多工作流历史会变成运行上下文。更多访问路径需要权限控制。更多 Agent 行动会产生记录，而这些记录又需要被日志化，用于合规、质量控制和未来复用。因此，越来越多的存储层将直接参与执行过程。

这就是为什么我们认为，企业 AI 的下一阶段最好被理解为一场数据架构转型。Agent 界面会吸引注意力，因为它是用户能看见的部分。但真正持久的操作性变化发生在底层：可信检索、身份、权限、审批、工具调用、写回、可观测性和保留。这里的结论是：Agent 需要企业上下文才能行动，但每一次行动又都会创造新的企业上下文。如果这些记录被保留、被治理并变得可检索，那么一个工作流的输出就会成为下一个工作流的输入。

**这个论点更令人不安的版本是：如果 Agentic AI 真的有效，企业存储栈其实建设不足。** Agent 会消费上下文、生成工作流记录、创造产物，并把过去的工作转化为未来的操作记忆。这意味着数据层必须同时支撑 Agentic 循环的两侧：在执行过程中快速访问温上下文，并对工作流创造的新数据进行持久保留。我们在此前的报告《Storage Wars》中讨论过硬件层面的这一点：闪存正在从持久化层变成推理内存层级中的主动延伸。Agentic 企业则把这个逻辑从 AI 机架延伸到了企业数据资产之中。

这一循环也为利益相关者提供了一种更有用的主题追踪方式。Agentic AI 支出不太可能以一个整洁的单独预算项出现。它会体现为服务工作、工作流模块、治理产品、可观测性和安全工具、数据平台现代化以及基础设施更新。最有利的公司，可能是那些最接近工作本身、最接近数据、也最接近 Agent 实际执行记录的公司。
