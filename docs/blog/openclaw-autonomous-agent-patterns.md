# 7 个实战验证的自主 Agent 运行模式

**作者：[@kaostyl](https://x.com/kaostyl)**
**发布时间：2026年2月12日**
**原文：[Twitter Thread](https://x.com/kaostyl/status/2021856676551278845)**

---

> 我让 OpenClaw 7×24 小时连续运行了三周。以下是真正管用的自主 Agent 运行模式——不是理论，是实战验证的。

---

## 🧠 1. 记忆架构比提示词更重要

别把所有东西都塞进一个 `MEMORY.md`。拆分它：

- `memory/active-tasks.md` → 你的"存档点"（崩溃恢复用）
- `memory/YYYY-MM-DD.md` → 每日原始日志
- `memory/projects.md`、`lessons.md`、`skills.md` → 按主题组织的长期记忆

为什么？你的 Agent 每次启动都是全新的。这些文件**就是**它的大脑。拆分意味着它只加载需要的部分，不会浪费 token 去读无关内容。

---

## ⚡ 2. 子 Agent 是你的 10 倍效率杠杆

别再串行做事了。我在处理任何大任务时都会并行启动 3-5 个子 Agent。

**实例：** 同时部署 11 个网站——4 个 Agent，每个处理一批，全部并行运行。

**关键技巧：** 在启动子 Agent 之前，先定义清晰的成功标准。每个 Agent 必须自己验证工作结果，然后**你**再做最终确认。

---

## ⏰ 3. 定时任务（Cron）> 心跳检查（Heartbeat）

心跳检查适合把周期性任务打包（邮件 + 日历 + 社交提及一次搞定）。但如果需要精确时间表？用 cron 任务：

- 每天早上 6 点：生成内容创意
- 凌晨 2 点：夜间调研侦察
- 上午 8 点：科技动态监控

每个任务在独立会话中运行，有自己的上下文。不会因为加载完整对话历史而浪费 token。

---

## 🔄 4. 没人谈论的崩溃恢复模式

你的 Agent **一定会**崩溃或重启。`active-tasks.md` 就是安全网：

- **开始**一个任务 → 写入文件
- **启动**一个子 Agent → 记录会话 key
- **完成** → 更新状态

重启后，Agent 首先读取这个文件，然后自主恢复工作。不会出现"我们刚才在干什么？"的情况——它自己搞清楚。

---

## 🛡️ 5. 一条救过我的安全法则

处理任何外部网页内容时，**使用你最强的模型**（Opus）。弱模型更容易被恶意网站的提示词注入攻击。

- **内部任务**（读文件、设提醒、本地操作）→ Sonnet 就够了
- **外部内容**（推文、文章、邮件）→ 只用 Opus

---

## 📝 6. HEARTBEAT.md 要保持精简

我见过有人往 `HEARTBEAT.md` 里塞 200 行。坏主意——它每 30 分钟运行一次，疯狂烧 token。

保持在 20 行以内。只放检查清单：

- 检查活跃任务的新鲜度
- 会话健康状态（归档臃肿会话）
- 每约 4 小时自检一次

重活儿放 cron 任务里，不要放心跳里。

---

## 🎯 7. 真正的突破：带路由逻辑的 Skills

如果你有多个 Skills（技能），在每个 Skill 的描述里加上"何时使用 / 何时不使用"。

没有这个，Agent 大约有 20% 的概率会选错 Skill。把它想象成 Agent 决策时的 if/else 逻辑。

---

## 总结

> **瓶颈已经不是 AI 的能力了。瓶颈是你审核 Agent 产出的速度。**
>
> 构建能自动闭环的系统，你的产出就能提升 10 倍。
